# Specification: Rebrand and Redesign as Awesome Slide

## Status

Planning

## Product Vision Alignment

Awesome Slide is a creator-first slide framework that keeps the openness and local-file workflow of open-slide while making the product feel like a polished app for creating, managing, and improving decks with agent assistance. The brand should feel modern, open-source friendly, creative, precise, and calm enough for sustained slide work.

This spec establishes the shared product identity used by the slide manager, in-app agent chat, connection settings, and theme library specs.

## Problem Statement

The project is still branded and visually structured as open-slide. That creates mismatch between the fork's future direction and the product experience users see in package names, docs, UI copy, CLI output, starter templates, marketing surfaces, and app chrome. The current identity also does not yet define a coherent UI system for the new product workflows: slide management, agent chat, model connections, and default themes.

## Goals

- Rename and rebrand the product as `awesome-slide` in technical contexts and `Awesome Slide` in user-facing contexts.
- Create a durable visual identity for the app, marketing site, docs, templates, and generated project surfaces.
- Use `references/REBRANDING_DESIGN_FINAL.md` as the primary visual direction: monochrome core UI, strong editorial type, large pastel color-block surfaces, light shadows, clear responsive rules, and the complete token front matter missing from the earlier `references/REBRANDING_DESIGN.md`.
- Incorporate `ui-ux-pro-max` guidance for data-dense editor workflows: drill-down layouts, keyboard access, visible focus states, responsive panels, skeleton/loading states, and efficient filtering/search.
- Keep the design appropriate for slide creation workflows: visual, spacious where slides need breathing room, and dense where users manage metadata or agent output.

## Non-Goals

- Do not redesign the slide authoring DSL or require existing slide source files to be rewritten.
- Do not introduce a closed-source brand or proprietary-only design asset dependency.
- Do not build the full slide management, agent chat, connection, or theme features in this spec; define the shared visual and naming foundation they rely on.
- Do not publish npm packages or change release automation as part of the planning artifact.

## User Stories

- As a slide creator, I want the app and docs to consistently say Awesome Slide so I know I am using the new product, not a lightly modified fork.
- As a first-time user, I want the starter project and marketing site to feel polished and current so I trust the framework for real presentations.
- As an open-source contributor, I want naming, package metadata, docs, and UI tokens to be consistent so contributions do not reintroduce old open-slide language.
- As a power user, I want a clean editor surface that keeps slide previews prominent while still supporting dense management controls.

## Functional Requirements

- FR-001: All user-facing product names must use `Awesome Slide` unless referring to a package, binary, folder, or command where `awesome-slide` is required.
- FR-002: The codebase must maintain a migration inventory for old `open-slide` references, grouped by package metadata, CLI output, docs, app UI, templates, examples, and internal compatibility aliases.
- FR-003: The package naming plan must decide which public package identifiers change immediately and which remain as compatibility aliases during migration.
- FR-004: The app shell must expose a shared Awesome Slide design token layer for color, typography, radius, spacing, border, shadow, focus, and motion.
- FR-005: The default UI theme must use a light canvas with high-contrast ink, restrained black primary actions, and pastel section or state surfaces derived from the rebrand reference.
- FR-006: The design system must define app-specific surfaces for slide browser, slide canvas, metadata panel, agent chat panel, connection status, theme cards, dialogs, command menus, and empty states.
- FR-007: The marketing site and docs must use the same brand vocabulary but may be more editorial and expressive than the app shell.
- FR-008: Starter templates generated by the CLI must use Awesome Slide branding in README files, sample copy, default config, and welcome slides.
- FR-009: Locale strings must be updated through the existing localization mechanism rather than hardcoding new labels inside components.
- FR-010: Any package rename that changes install or import behavior must include migration notes and compatibility acceptance tests.
- FR-011: Implemented visual tokens must preserve the final reference palette: black/white primary system, exact pastel block family, and magenta as a scarce marketing promo accent. Vermillion/red accents or a dark-default marketing shell require explicit design approval and must otherwise be treated as drift to fix.

## UX Requirements

- UX-001: The visual system must feel like a creative editor, not a generic admin dashboard.
- UX-002: The app should use a monochrome base for navigation, panels, body text, primary actions, and dense controls.
- UX-003: Pastel color blocks should be reserved for major context shifts, onboarding, empty states, theme showcases, and celebratory moments, not every card.
- UX-004: Use open-source font substitutes from the reference: Inter or Geist for the main sans family and JetBrains Mono or Geist Mono for mono labels.
- UX-005: Keep app controls compact and predictable: icon buttons for repeated tools, segmented controls for modes, toggles for binary settings, menus for option sets, and text buttons for clear commands.
- UX-006: Do not use emoji as UI icons. Use a consistent icon family such as Lucide where icons are needed.
- UX-007: All clickable controls must show pointer cursor, hover feedback, and visible keyboard focus.
- UX-008: Support responsive breakpoints for mobile, tablet, desktop, and wide desktop, with no horizontal scroll for standard app views.
- UX-009: Loading states must use skeletons or progress indicators that preserve layout and avoid blank panels.
- UX-010: Motion must be subtle, 150-300ms for interface transitions, and respect reduced-motion preferences.

## Technical Considerations

- Main implementation areas are expected to include `packages/core` for runtime UI, `packages/cli` for scaffolding and templates, `apps/demo` for dogfooding, and `apps/web` for marketing/docs.
- The current app already has slide browser, folder, and theme surfaces in `packages/core/src/app/routes/home.tsx`, `packages/core/src/app/lib/folders.ts`, and `packages/core/src/app/components/themes/themes-gallery.tsx`; the rebrand should preserve useful behavior while replacing visual language.
- `packages/core/src/app/components/ui` is shadcn-generated and should not be hand-edited unless regenerated.
- If public package names change, compatibility should be explicit: old package imports either remain supported for one major cycle or fail with actionable migration guidance.
- Because changes to `packages/core` or `packages/cli` require a changeset, implementation work for this spec must add one when code changes begin.
- The design system should avoid becoming a one-note dark slate UI even though the UI guidance search identified a data-dense dashboard pattern. Awesome Slide should apply that density to interaction structure, not to a dark dashboard aesthetic.
- Brand tokens should be documented in a durable design source such as `DESIGN.md` or an equivalent repo-local design-system document before broad UI implementation.

## Acceptance Criteria

- AC-001: A brand inventory identifies every `open-slide` user-facing reference and the intended new value or compatibility decision.
- AC-002: The product has documented naming rules for `Awesome Slide`, `awesome-slide`, package identifiers, CLI commands, config names, and legacy compatibility labels.
- AC-003: A design system document defines colors, typography, spacing, radius, component treatments, motion, responsive behavior, and accessibility rules.
- AC-004: The core app shell, demo, CLI template, and web/docs surfaces all use the new brand language.
- AC-005: Old user-facing labels do not appear in standard app flows unless explicitly marked as compatibility or migration text.
- AC-006: UI contrast, keyboard focus, responsive layout, and reduced-motion behavior are covered by review or automated checks for the redesigned surfaces.
- AC-007: Existing slide projects can still load after the rebrand migration.
- AC-008: Changes touching `packages/core` or `packages/cli` include an appropriate changeset.
- AC-009: The final validation notes include a design-alignment audit comparing implemented runtime, docs, and marketing surfaces against `references/REBRANDING_DESIGN_FINAL.md`.

## Implementation Phases

### Phase 1: Brand and Migration Audit

- Inventory names, package metadata, CLI output, docs, locale strings, templates, screenshots, tests, and config examples.
- Decide package and command compatibility strategy.
- Document migration rules and user-facing terminology.

### Phase 2: Design System Foundation

- Create the Awesome Slide design source of truth.
- Define tokens for app shell, marketing/docs, slide management, agent chat, connection settings, and theme gallery.
- Map the rebrand reference and UI guidance into concrete component rules.

### Phase 3: App and Template Rebrand

- Update app chrome, loading states, empty states, theme gallery, slide browser, and starter template copy.
- Update CLI scaffold output and generated project README content.
- Add compatibility notices where old names still appear by design.

### Phase 4: Marketing and Documentation Refresh

- Redesign docs and marketing surfaces around the new visual identity.
- Add migration guidance from open-slide to Awesome Slide.
- Update screenshots or generated previews after the app redesign stabilizes.

### Phase 5: Validation and Release Prep

- Run typecheck, tests, and Biome checks.
- Verify accessibility and responsive behavior at 375px, 768px, 1024px, and 1440px.
- Add changesets for touched packages and prepare release notes.
